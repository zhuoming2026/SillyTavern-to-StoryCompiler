# Characters-to-world — 故事构建器

## 职责边界

**在角色卡导入完成后接手（依赖角色卡导入的三个产出文件）：**

- `设定/角色卡原始数据.json`
- `设定/世界书概览.md`（用户已打勾）
- `设定/🎬 开场白.md`

**不做的事情：**
- ❌ 不提取原始数据（那是 importer 的活）
- ❌ 不做世界书概览（那是 importer 的活）
- ❌ 不替用户判断——展示、提问、等回答

---

## 工作流

### Step 0：环境检测

Characters-to-world 所有步骤均为纯文件操作（读 JSON → 分析 → 写设定文件），**不依赖 bash 或脚本执行**，在所有平台通用。

无需特别适配。如果 `bash` 不可用，不影响本技能执行。

---

## 工作流

### Step 3：读取用户勾选 → 生成全部设定文件

> ⚡ **角色卡导入后的衔接规则：**
> - `SillyTavern-Characters-importer` 已提取了角色卡原始数据并生成了 `设定/` 下的全部文件
> - **不需要重跑 3a-3e**（硬提取 + 骨架归纳已在导入器完成）
> - 直接从 **3f** 开始（检测 MVU 并生成状态文件）
> - 故事名使用 `设定/角色卡原始数据.json` 中的 `name` 字段
> - 如果 `故事/rp-{name}/` 已存在 → 加 `-1` `-2` 后缀

#### 3a. 读取用户勾选

读取 `设定/世界书概览.md`，解析 `[x]` 标记的条目 ID。
从 `设定/角色卡原始数据.json` 中提取对应的完整条目内容。

如果用户还没打勾 → 提示用户先打勾再继续。

#### 3b-1. 硬提取（脚本执行，不经过 AI）

根据条目的 `comment` 和 `keys` 将勾选条目原样写入对应文件：

| 分类 | 判断依据 | 输出文件 |
|:-----|:---------|:---------|
| 世界观 | 涉及场景/系统/设定/规则/玩法 | `设定/世界背景.md`（原始草稿） |
| 角色 | 涉及角色卡/人物描述 | `设定/角色.md`（原始草稿） |
| 主角 | 涉及主角/玩家角色 | `设定/主角.md` |
| 风格/变量 | 风格指南条目 / MVU 变量条目 | 单独处理 |

**同时产出硬数据文件（模式匹配，不经过 AI）：**
- `设定/角色硬数据.md` — 名称、年龄、外貌、固定关系等可模式匹配的事实

**规则：** 硬提取只做原样复制，不做内容修改或 AI 补充。
**注意：** 原始草稿文件内容可能是混合的（一个条目同时包含世界观和角色信息），这是正常的——下一层 AI 骨架归纳会进行拆分重组。

#### 3b-2. AI 骨架归纳（有固定结构约束）

> 此步骤在硬提取完成后执行，目的是将混合的原始条目拆分为结构化的设定文件。
> AI 在固定骨架内填充内容，不得擅自增减章节，不得删除原文中的任何事实。

**输入：** 3b-1 产出的原始草稿文件 + `设定/角色卡原始数据.json`
**锚点要求：** 每项归纳结果必须引用原文行号（见下方示例）

**输出文件与固定骨架：**

| 文件 | 固定骨架 | 说明 |
|:-----|:---------|:------|
| `设定/世界背景.md`（最终版） | 世界类型/时代背景 → 核心规则/特殊设定 → 势力/家族/组织 → 当前时间点/事件起点 | 覆盖原始草稿 |
| `设定/角色.md`（最终版） | 每个角色：名称/身份 → 外貌特征 → 性格特质(引用行号) → 动机/目标 → 关系 → 原文附录 | 覆盖原始草稿 |
| `设定/主角.md`（最终版） | 名称/身份 → 玩家可定 → 故事已固定 → 多层性格(如有) | 覆盖原始草稿 |

**行号引用示例（必须在每个归纳项中标注来源）：**

```markdown
## 角色：张三

- **身份**：新手冒险者，铁匠之子（来源：设定/角色卡原始数据.json #12）
- **性格**：热心肠但有点冒失，容易说错话（来源：设定/角色卡原始数据.json #15-18）
  - 原文摘要："张三是个热心肠的小伙子，就是嘴上没把门的……"
- **多层性格**（如果存在）：
  | 层面 | 核心特征 | 激活条件 |
  |:-----|:---------|:---------|
  | 日常状态 | 开朗健谈、乐于助人 | 默认启用 |
  | 战斗状态 | 专注认真、话少 | 进入战斗后 |
  | 醉酒状态 | 胡言乱语、抱着人哭 | 喝酒后 |
```

**约束规则：**
1. 不得删除原始数据中的任何事实（可以在骨架中重组，不得删除）
2. 不得擅自增减骨架章节（固定结构不可改动）
3. 每条归纳必须标注来源行号
4. AI 推断的内容必须标注"（AI 推断）"以区别于原文
5. 如果原始数据无法填充骨架中的某个字段 → 留空并标注"（原始数据未提供）"

**关于多层性格：** AI 从原始数据中识别角色是否有多个性格层面（如人前 vs 人后、前期 vs 后期、特定关系下的特殊态度）。如果有，在角色条目中列出层面表；如果只有一个层面，则不需要。

#### 3c. 提取风格指南

**优先级：**
1. 用户勾选的条目中，涉及「文风/叙事/描写/风格/用语」的条目（最高优先）
2. 开场白 `first_mes` 的语言风格（次优先）
3. `mes_example` 的对话风格（补充参考）

**没有文风条目时**，从开场白中提取风格特征：
- 叙事基调与氛围
- 叙事视角
- 语言风格特征（句式、修饰、感官侧重）
- 各角色语言特征（从对话中提取典型口吻和用词）
- 来源说明

**产出：** `设定/风格指南.md`

#### 3d. 提取变量系统（MVU 系统 ★）

**核心入口：** 检查用户勾选的条目中，`comment` 含「更新规则」且 `content` 含 `<UpdateVariable>` 的条目。

如果存在，提取完整的 MVU 系统到 `设定/📊 状态变量.md`：

```
| 变量 | 说明 | 类型 | 更新规则 |
```

包含：
- 变量定义表
- 序列等级对照表（如有）
- 强制更新法则（逐条提取，原样保留）
- 输出格式示例（_.set() 调用格式）

⚠️ **特别注意：** MVU 系统的核心是 `<UpdateVariable>` + `_.set()` + 强制更新法则。提取时保留完整的格式规范（纹章 JSON 嵌套格式、数值纯数字要求等）。

如果找不到 MVU 条目 → 空文件，标注"未检测到变量系统"。

#### 3e. 提取思维链（如果有）

**核心入口：** 检查勾选的条目中是否有涉及「思考步骤/写作流程/叙事步骤/内部流程」的内容。

如有 → 提取并写入 `设定/思维链.md`
如无 → **不生成空文件**，AGENTS.md 中使用通用模板

#### 3f. 提取 MVU 状态数据 + 状态栏（★ V2.2 增强版）

> **目标：** 从角色卡中提取 MVU 相关资产，生成 `状态/stat_data.js`（结构化数据）和 `状态/状态栏.html`（可视化展示）。
> **关键设计：** 本工具不同于 SillyTavern 运行时自动解析——我们由**用户手动执行 `#更新状态`** 来维护变量。所以开场白多值、缺失值等问题通过**用户交互**解决。

> ⚡ **如果角色卡没有 MVU 组件：跳过整个 Step 3f，3f-1 检测后直接结束。不生成 stat_data.js、状态数据.md、状态栏.html。**

##### 3f-1. 检测 MVU 组件

角色卡中 MVU 的四大组件及其检测位置。**如果 4 项全无 → 跳过 Step 3f（此段全部跳过）：**

| 组件 | 检测位置 | 判断依据 |
|:-----|:---------|:---------|
| 🏗️ **MVU 引擎** | `extensions.tavern_helper` 中的脚本 | 含 `MagVarUpdate` 且含 `bundle.js` |
| 📐 **Zod Schema** | `extensions.tavern_helper` 中的脚本 | 含 `registerMvuSchema` |
| 📦 **initvar 数据** | 多处（见下文） | 见 3f-2 |
| 🖥️ **状态栏 HTML** | `extensions.regex_scripts[].findRegex` | 含 `StatusPlaceHolderImpl` |

**注意 JSON 结构变体：** `tavern_helper` 可能是两种格式之一，都需要支持：
- **dict 格式：** `{"scripts": [...], "variables": {...}}`（最常见）
- **数组对格式：** `[["scripts", [...]], ["variables", {}]]`（部分 SillyTavern 导出）
- 两种格式内容相同，只是结构不同。检测时遍历所有脚本内容，不依赖具体路径。

**注意脚本 `name` 的多样性：** 实际角色卡中脚本名称不固定，可能是 `MVU`、`MVUbeta`、`变量结构`、`变量定义`、`mingyue-mvu-xxx` 等。**不要依赖 `name` 字段**，以上述 content 特征为准。

**注意状态栏可能有双重正则：** 同一个 `<StatusPlaceHolderImpl/>` 可能被多个正则脚本处理：
- 一个 `promptOnly: true` 的脚本负责**对 AI 隐藏**（替换为空字符串）
- 另一个脚本负责**渲染 HTML 状态栏**给用户看
- 提取时以渲染用的那个为准（含 `replaceString` HTML 的脚本）

**检测结果输出：**
```markdown
[检测结果]
- MVU 引擎: ✅/❌
- Zod Schema: ✅ (复杂度: 低/中/高)
- initvar 数据: ✅ (来源: 世界书/开场白/两者)
- 状态栏 HTML: ✅ (风格: xxx)
```

##### 3f-2. 提取 initvar（多来源）

initvar 数据可能出现在多个位置，按优先级依次检查：

**来源 A：世界书条目**
- 扫描所有世界书 `entries[]`（无论 `enabled` 与否）
- 查找 `comment` 含 `[initvar]` 或 `记录初始化` 或 `变量初始化` 的条目
- 读取 `content` 中的 JSON5/YAML 初始值数据

**来源 B：开场白消息体**
- 检查 `data.first_mes` 和 `data.alternate_greetings[]`
- 查找其中包含 `<UpdateVariable>` 标签的块
- 提取标签内的 `_.set()` 指令，解析为变量树

**来源 C：EJS 模板**
- 检查所有消息体中含 `<% setvar(` 的 EJS 语法
- 如有 → 标注"含 EJS 动态初始化，需注意"

**处理多开场白：**

检测到多个开场白（1 个 first_mes + N 个 alternate_greetings）且各自有不同初始值时：

```
步骤：
1. 列出所有可用开场白（编号 + 前 30 字摘要）
2. 询问用户：**"检测到多个开场白各有不同初始值，请选择使用哪个开场白？"**
   - 选项 1: [开场白 A 摘要...]
   - 选项 2: [开场白 B 摘要...]
   - 选项 N: ...
   - 或：让我自定义基础值
3. 根据用户选择，提取对应开场白中的 initvar
```

> 如果只有一个开场白（仅 first_mes），自动使用，无需询问。

**引导用户确认基础值：**

提取初始值后，展示给用户确认：

```markdown
[已提取基础变量]
| 变量路径 | 初始值 | 说明 |
|:---------|:-------|:-----|
| 世界状态.当前时间 | 第三日·午后 | 来自 initvar |
| 旅人.体力 | 100 | 范围 0~100 |
| ... | ... | ... |

请检查以上基础值：
- ✅ 没问题，直接使用
- ✏️ 我要修改某些值 → 用户指定修改项
- ❌ 我要自己重新设置 → 跳过自动生成
```

##### 3f-3. 提取 Zod Schema（如果有）

从角色卡 JSON 的所有脚本位置中查找。注意 `tavern_helper` 可能是 dict 或数组对格式，遍历所有能找到的脚本：

- **判断依据：** `content` 含 `registerMvuSchema`（不论 `name` 字段值，不论 JSON 结构）
- **找到** → 提取 `content` 中的 Schema 定义代码
- **找不到** → 标记"无 Zod Schema"

**分析 Schema 结构（给 AI 理解用）：**
- 顶级路径有哪些（如 `世界环境`、`主角`、`契约者`）
- 每个路径的类型（`z.string()` / `z.number()` / `z.enum([...])` / `z.record()`）
- 约束条件（`.range()`、`.clamp()`、`.describe()`、`.prefault()`）
- 动态键（`z.record(z.string(), z.object({...}))` → 键名不固定）

##### 3f-6. 生成 `状态/stat_data.js`

将用户确认后的 initvar + Schema 合并为 single-source-of-truth 数据文件。

**生成规则：**
- 变量路径从 initvar 的嵌套结构推导
- 变量类型和范围从 Zod Schema 提取为注释（如有）
- 初始值使用用户确认后的值
- `{{user}}` 占位符保留不替换
- **无 Schema fallback：** 如果角色卡无 Zod Schema，仅使用 initvar 结构生成，类型标注为 `unknown`

```javascript
// 状态数据 — 由 Characters-to-world 从角色卡自动生成
// 每次写作后通过 #更新状态 指令同步更新
// 来源: [initvar]（来自世界书条目）+ Zod Schema（如有）
// ⚠️ 首次使用前请用 #更新状态 确认初始值

const statData = {
  "世界状态": {
    "当前时间": "第三日·午后", // string
    "所处地点": "边境小镇"     // string
  },
  "旅人": {
    "体力": 100,              // number (0~100)
    "心情": "平静",           // string
    "装备": "旅行者轻甲",    // string
    "背包": {
      "金币": 50,
      "干粮": "3日份"
    }
  }
};
```

**无 Zod Schema 时的 fallback：**
- 从 initvar JSON5 的值的 JS 类型推断（string / number / boolean / object / array）
- 不生成类型注释（标注 `// 类型自推断，无 Schema`）
- 如果是嵌套对象，保留嵌套结构

**同时生成 `状态/状态数据.md`**（同一份数据，Markdown 表格格式供 AI 阅读）：

```markdown
# 状态数据

> 来源：[initvar] + Zod Schema
> 更新方式：`#更新状态`

| 变量路径 | 当前值 | 类型 |
|:---------|:-------|:-----|
| 世界状态.当前时间 | 第三日·午后 | string |
| 世界状态.所处地点 | 边境小镇 | string |
| 旅人.体力 | 100 | number |
```

##### 3f-7. 生成 `状态/状态栏.html`（如有状态栏 HTML 模板）

将提取到的状态栏 HTML 模板转换为脱酒馆版本：

**完整 API 替换清单：**

| 酒馆 API | 替换方案 | 说明 |
|:---------|:---------|:-----|
| `getAllVariables()` | 改为从 `statData` 读取 | 🔴 必须 — 获取全部变量 |
| `getChatMessages()` / `getCurrentMessageId()` | 删除（状态栏是静态展示） | 🔴 必须 — 不需要实时楼层数据 |
| `window.TavernHelper.getVariables(...)` | 替换为 `statData` | 🔴 必须 — 获取变量 |
| `waitGlobalInitialized('Mvu')` | 删除 | 🔴 必须 — 不需要等待 MVU 加载 |
| `eventOn(Mvu.events.VARIABLE_UPDATE_ENDED, ...)` | 替换为 `document.addEventListener('DOMContentLoaded', ...)` | 🔴 必须 — 不再监听实时事件 |
| `Mvu.events.*` | 全部删除 | 🔴 必须 — 不再依赖 MVU 事件 |
| `_.get(obj, path, default)` | `safeGet(obj, path, default)` | 🟡 建议 — lodash 替换 |
| `$('#id').text()` / `$('#id').css()` | `document.getElementById().textContent / style` | 🟡 建议 — jQuery 替换 |
| `$('.class')` | `document.querySelectorAll()` | 🟡 建议 |
| `toastr.success/error/warning` | 删除或 console.log | 🟢 可优化 — 通知提示 |
| SillyTavern CSS 变量/字体 | 内联 fallback 值 | 🟢 可优化 — 样式依赖 |
| 立绘路径 `characters/xxx.png` | 转换为本地相对路径 | 🟢 可优化 — 图片资源 |

**生成的 HTML 结构：**
```html
<script src="stat_data.js"></script>
<script>
/* 从 statData 读取数据，不依赖酒馆运行时 */
function safeGet(obj, path, fallback) {
  var parts = path.split('.');
  var cur = obj;
  for (var i = 0; i < parts.length; i++) {
    if (cur == null || typeof cur !== 'object') return fallback;
    cur = cur[parts[i]];
  }
  return cur !== undefined && cur !== null ? cur : fallback;
}
function populate() {
  var d = typeof statData !== 'undefined' ? statData : {};
  // 按状态栏原始设计映射 DOM 元素
  document.getElementById('val-time').textContent = safeGet(d, '世界状态.当前时间', '未知');
  // ...
}
document.addEventListener('DOMContentLoaded', populate);
</script>
```

> 如角色卡无状态栏正则 → **不生成 HTML 文件**

##### 3f-7b. AI 校验状态栏 HTML（★ 质量保障）

生成 `状态/状态栏.html` 后，必须进行语法校验，避免浏览器报错。

> 此步骤也可作为独立 skill 运行：`技能/statusbar-validator/SKILL.md`
> 对已生成的文件可直接说"校验状态栏 [路径]"调用。

**校验清单（逐项检查）：**

```markdown
□ 1. 花括号 {} 是否配对（打开/关闭数量一致）
□ 2. 圆括号 () 是否配对（特别是 function 调用和 innerHTML 赋值）
□ 3. 无残留酒馆 API（getAllVariables、waitGlobalInitialized、eventOn、Mvu.、TavernHelper）
□ 4. 无残留 jQuery $() 调用（应全部替换为 document.querySelector/getElementById）
□ 5. 无残留 .html() 调用（应全部替换为 .innerHTML =）
□ 6. 无孤立的 ); 残留（原 `$('#xxx').html(...)` 替换后多余的闭合括号）
□ 7. `<script src="stat_data.js"></script>` 引用是否存在
□ 8. statData 变量是否采用 `var` 声明（而非 const，确保模块脚本可访问）
□ 9. DOM 操作在 `DOMContentLoaded` 事件后进行
```

**校验方法：** 通读生成的 HTML 文件中的 JavaScript 代码，逐项核对上述清单。

**发现问题的处理：**
- 语法错误 → 立即修复，重新校验
- 逻辑错误（如数据路径不匹配）→ 调整变量引用路径
- 样式问题（如布局错乱）→ 可保留，不影响功能

**校验通过标准：** 9 项全部 ✅，状态栏可直接在浏览器打开不报错。
