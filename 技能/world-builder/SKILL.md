# world-builder — 故事构建器

## 职责

将输入源（角色卡PNG/JSON、小说导入、用户对话）转化为**可直接扮演的完整故事包**。核心产出是每个故事独一无二的 `AGENTS.md`。

## 输入源

world-builder 可接受以下任意一种或多种输入：

| 来源 | 典型场景 | 提供的原始数据 |
|:-----|:---------|:---------------|
| `SillyTavern-Characters-importer` 输出 | 用户说"解析这张角色卡" | `raw_data.json`（含角色/世界观/系统指令/开场白） |
| 用户直接对话 | 用户说"我要写一个修仙故事" | 对话文本 |
| `novel-import` / `novel-content-extraction` | 用户说"导入这本小说" | 结构化小说数据 |

## 工作流

### Step 1：初始化故事目录

```
输入：故事名（如"修仙"） + 原始数据（可选）
    ↓
创建：故事/rp-修仙/
    ↓
从 模板/故事/ 复制骨架（子目录结构）
    ↓
如果有原始数据，写入 基础数据/raw_data.json
```

### Step 2：读取并合并输入

如果有原始数据（来自角色卡或小说导入），从中提取：

```
原始数据 JSON
    ↓
提取：
  ├── 主角信息       → 临时变量
  ├── NPC 角色        → 临时变量
  ├── 世界观描述      → 临时变量
  ├── 开场白          → 临时变量（first_mes / alternate_greetings）
  ├── 系统指令/规则   → 临时变量（system_prompt）
  └── 世界书条目      → 临时变量（character_book.entries）
```

如果用户直接对话，则按原有引导流程（Step 3）逐项询问。

### Step 3：校验完整性

检查 `故事/rp-故事名/` 目录，逐项报告：

| 检查项 | 必需？ | 判定 |
|:-------|:-------|:------|
| `设定/世界背景.md` 有内容 | ✅ | 缺失则引导用户填写，可参考 `通用知识/世界观/` |
| `设定/角色.md` 有内容 | ✅ | 缺失则从原始数据提取或引导，可参考 `通用知识/角色原型/` |
| `设定/主角.md` 有内容 | ✅ | 缺失则从原始数据提取或引导 |
| `设定/开场白.md` 有内容 | ✅ | 缺失则从原始数据提取或引导 |
| `设定/风格指南.md` 有内容 | ✅ | 缺失则引导用户确定文风方向，可参考 `通用知识/文风/` |
| `设定/状态变量.md` | ⚠️ 可选 | 检测原始数据中是否有变量系统，或参考 `通用知识/系统机制/` |
| `风格/` 有文风文件 | ⚠️ 可选 | 提示用户可调用 style-trainer 或引用 `通用知识/文风/` |

**输出校验报告：**
```
📋 故事「修仙」完整性检查：

✅ 世界背景 — 已从角色卡提取
✅ 角色 — 3个角色已提取
✅ 主角 — 已命名
✅ 开场白 — 2个开场白选项
❌ 风格指南 — 缺失！需要确定文风方向
⚠️ 状态变量 — 未检测到变量系统（可选）

是否补全缺失项？[y/n]
```

### Step 4：补全缺失项

对每个缺失项，按以下方式处理：

#### 4a. 从原始数据自动提取

如果输入源是角色卡或小说，尽量自动填充：
- 从 `data.character_book.entries` 提取世界观 → `世界背景.md`
- 从 `data.name` / `data.description` 提取角色 → `角色.md` + `主角.md`
- 从 `data.first_mes` / `data.alternate_greetings` 提取 → `开场白.md`
- 从 `data.system_prompt` 提取扮演规则 → 用于 AGENTS.md 生成

#### 4b. 引导用户填写

无法自动提取时，按以下顺序引导用户（保留原有的逐项引导流程）：

1. **世界背景** — 世界观类型、社会结构、当前状态
2. **角色** — 主要角色名单、核心特征
3. **主角** — 名字、身份、特质
4. **开场白** — 故事从哪开始
5. **风格指南** — 基调、视角、文风方向
6. **选开内容** — NTR/NTL、近亲DLC、状态栏等

### Step 5：规则定制

根据故事内容和角色卡数据，自动定制扮演规则：

#### 5a. 检测状态变量系统

检查原始数据中是否有变量相关的内容：
- 角色卡 JSON 中是否有 `state`、`好感度`、`变量` 等字段
- 开场白中是否有 `<Status_block>` 或类似结构
- 用户是否明确说有变量系统

如果有，生成 `设定/状态变量.md`，并在 AGENTS.md 中包含状态系统章节。

#### 5b. 提取特殊规则

从 `data.system_prompt` / `data.post_history_instructions` 中提取：
- 特殊的行为约束
- 世界专属规则
- 内容边界

这些规则将嵌入到 AGENTS.md 的「核心规则」章节中。

### Step 6：生成 AGENTS.md（核心产出）

这是最关键的一步。从 `模板/故事/AGENTS.md` 读取标准模板，然后填充各章节：

| AGENTS.md 章节 | 内容来源 |
|:---------------|:---------|
| 你是谁 + 核心规则 | 固定模板 + 从 system_prompt 提取的特殊规则 |
| 启动应答 | 固定模板 + 故事名/进度 |
| 故事设定 | 从 `设定/世界背景.md` 读取 |
| 角色设定 | 从 `设定/角色.md` + `设定/主角.md` 读取，格式化 |
| 状态系统 | **如果**存在 `设定/状态变量.md`，动态生成此章节 |
| 写作风格 | 从 `设定/风格指南.md` 读取 |
| 核心写作规则 | 固定模板 + 从用户选择的方向定调生成 |
| 启动菜单 | 固定模板 + 从 `设定/开场白.md` 读取开场白选项 |
| 默认开场白 | 嵌入 `设定/开场白.md` 全文 |
| 聊天历史系统 | 固定模板 |
| 回复格式 | 固定模板 |
| 禁止事项 | 固定模板 |

**生成方式**：依次读取各设定文件的内容，逐章写入 AGENTS.md，替换模板中的 `{{占位符}}`。

### Step 7：生成支持文件

| 文件 | 生成方式 |
|:-----|:---------|
| `故事绑定.md` | 从模板复制 + 填入故事名 |
| `README.md` | 从模板复制 + 填入故事名和简介 |
| `元数据/故事元数据.md` | 自动生成（当前日期、状态=设定中） |
| `状态/状态机.md` | 从模板复制 |
| `状态/世界概览.md` | 从 `世界背景.md` 提取一句话 |
| `状态/角色状态.md` | 从 `角色.md` + `主角.md` 生成初始状态 |
| `状态/事件日志.md` | 空模板 |
| `配置/扮演规则.md` | 从模板复制（指向 AGENTS.md） |

### Step 8：输出确认

向用户展示最终产出：

```
✅ 故事「修仙」构建完成！

📦 故事/rp-修仙/
├── AGENTS.md              ← 扮演规则（独一无二）
├── 故事绑定.md            ← 锁文件
├── README.md
├── 设定/                   ← 世界观/角色/开场白
├── 状态/                   ← 进度追踪
├── 配置/                   ← 扮演规则
├── 写作/ 风格/ 基础数据/

🎭 你可以：
- 直接进入此故事开始扮演
- 先调用 style-trainer 训练文风
- 继续修改设定

说"进入 修仙"开始！
```

## 标准扮演模板

`模板/故事/AGENTS.md` 是标准扮演模板，包含固定章节结构和占位符。world-builder 通过读取各设定文件来填充占位符，生成独一无二的 AGENTS.md。

标准模板的章节结构固定为：

1. 你是谁
2. 启动应答
3. 故事设定
4. 角色设定
5. 状态系统（可选章节）
6. 写作风格
7. 核心写作规则
8. 启动菜单
9. 默认开场白
10. 聊天历史系统
11. 回复格式
12. 禁止事项

每个故事的 AGENTS.md 都基于此模板，但内容完全不同。
