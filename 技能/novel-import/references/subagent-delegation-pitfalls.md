# 子代理委托踩坑记录

## 问题：mimo 等较小模型通过 delegate_task 委托时不执行实际工作

### 症状
- 子代理返回极快（3-20秒）
- 只做了 1-3 次 API 调用
- 返回摘要类似 "我将开始读取文件..." 但没有实际读取或写入
- `find` 检查输出目录无任何文件生成

### 原因
较小模型（如 mimo-v2.5-pro）在 delegate_task 上下文中：
1. 可能将 goal 理解为"准备做"而非"立即执行"
2. 工具调用能力在子代理隔离环境中受限
3. 缺乏足够的推理深度来执行多步骤任务（读文件→分析→写多个文件）

### 解决方案
1. **首选**：不委托，直接在主会话中用 execute_code 或终端处理
2. **次选**：将任务拆分为更小的单步任务（每个子代理只做一件事：读文件+写一个文件）
3. **验证方法**：子代理返回后立即检查输出文件是否存在

### 何时可以委托
- 模型为 Claude/GPT-4 级别
- max_concurrent_children ≥ 3
- 任务简单明确（单步：读X写Y）

## 大文件处理策略选择

| 文件大小 | 推荐策略 |
|---------|---------|
| < 500KB | 直接在主会话中分析 |
| 500KB-2MB | 切分章节 → 程序化预提取 → 主会话读精简版分析 |
| > 2MB | 切分 → 程序化提取 → 精简 → 分批委托（仅限强模型） |

## 并发限制

- delegate_task 的并发数由 `delegation.max_concurrent_children` 控制
- 默认值可能为 1 或 3，需检查 config.yaml
- 如果并发为 1，并行委托会回退为串行
